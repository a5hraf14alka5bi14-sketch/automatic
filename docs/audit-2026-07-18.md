# Codebase Audit — Automatic Restaurant OS

**Date:** 2026-07-18
**Scope:** Full repository review (architecture, security, database, tests, CI/CD, dependencies, code quality)
**Method:** Direct inspection of every route, migration, and config file; live verification against a fresh local PostgreSQL 16 instance; `npm audit`; secret scanning; cross-referencing against the 2026-07-05 Full Project Review and this repo's own `.agents/memory/` notes.

---

## Overall Verdict

This is a **well-engineered, actively-hardened codebase**. The headline finding of this audit is good news: **every item (P1–P5) from the 2026-07-05 Full Project Review has already been fixed in the code** — role guards on sensitive endpoints, `FOR UPDATE` locking, unique constraints, `NUMERIC(10,3)` precision, Joi validation on every route, dynamic `import()` for jsPDF, real error handling in every `catch` block, and full test-coverage tooling. None of this was reflected in the Notion Remediation Backlog, which still lists all 13 items as "Not Started" — that tracker needs a sync pass, not the code.

The real issues found this session are a **different, narrower pattern**: schema and config drift between the long-running Replit database and what's actually committed to git. Every instance was masked for months because the Replit database never started from empty — it only became visible when this repo's CI was, for the first time, exercised against a genuinely fresh environment (see the parallel PR that fixed CI end-to-end).

## ✅ Security — P1 items (2026-07-05 review), verified fixed

- `GET /api/menu/food-cost` and `/api/menu/recipe/link-summary` — now `requireRole('admin', 'manager')`
- `GET /api/inventory/stats` and `/movements` — now `requireRole('admin', 'manager')`
- Order financial fields (`subtotal`, `tax`, `total`, `discount`, `payment_method`, `void_reason`, etc.) — stripped server-side for `kitchen`/`staff` roles via `filterOrderFields()`
- `must_change_password` — re-checked from the database on every `/refresh` call, not just at login; an in-flight session can no longer bypass a mandated password change

## ✅ Database integrity — P2 items, verified fixed

- `POST /purchase-orders/:id/receive` now takes `SELECT ... FOR UPDATE` before mutating stock
- `recipe_ingredients` has `UNIQUE (menu_item_id, inventory_item_id)` (migration 010)
- `finance_entries.amount` and `modifier_groups.additional_price` are `NUMERIC(10,3)` (migration 010)

## ✅ Backend validation — P3 items, verified fixed

- `shifts.js`, `suppliers.js` (PO routes), and `users.js` all route through the shared `validate()` Joi middleware

## ✅ Frontend — P4 items, verified fixed

- `jspdf` / `jspdf-autotable` are dynamically `import()`ed in `exportUtils.js`, not bundled into the initial load
- No empty `catch {}` blocks remain in `POS.jsx` — every one calls `showToast(...)` or otherwise surfaces the failure

## ✅ Test coverage — P5, verified fixed

- `vitest` coverage is fully configured (`@vitest/coverage-v8`, text/html/lcov reporters, `npm run test:coverage`)

- **~24,000 lines** across `server/` (9.2k), `src/` (15.3k) — plus **9.6k lines of tests**, an unusually high test-to-code ratio for a project this size

This level of discipline (structured logging via `server/logger.js`, consistent error handling, a real migration runner with advisory locking, a custom secret scanner wired into pre-commit + CI) is well above what's typical for a project at this stage.

## 🟡 Remaining / open items

1. **Recover the real `.config/.semgrep/semgrep_rules.json` from Replit** and commit it, then revert CI off the public `p/security-audit` stand-in.
2. **Audit for more "exists live, never committed" drift.** Three separate instances were found in one session (semgrep config, package-lock registry, five DB columns) purely by trying a fresh install — there's a reasonable chance more exist. A clean way to check going forward: any time a fresh `npm ci` + `node scripts/init-db.js` + `npm test` doesn't pass on a brand-new database, treat it as a drift signal, not a flake.
3. **Sync the Notion Remediation Backlog** — it should mark all 13 P1–P5 items done; right now it says the opposite of what the code shows.
4. **Outdated major-version dependencies** (Capacitor 7.x → 8.x, Electron 41 → 43, bcryptjs 2.x → 3.x, concurrently 8.x → 10.x) — `npm audit` is clean, so this is routine maintenance, not urgent.
5. **No systematic check that `validators.js` field names match real columns** — the `password_hash` bug (#6 above) is exactly this class of bug; a small script cross-referencing Joi schema keys against `information_schema.columns` would catch this category automatically.
